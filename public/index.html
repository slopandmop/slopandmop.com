Source DD EasyEnclave
Compose slop · slap it on a VM · mop up

Your own sysadmin, running inside a sealed enclave.

Click the button. We boot an Intel TDX VM and start claude inside it. Tell it what you want — /deploy-openclaw, say — and it spins up the VMs, wires the ingress, and hands you a URL. Auto-mop in 6 hours.

No sign-in. One free sm per source IP per day. You'll land in a terminal-in-browser chat. 

        

      


      

        

          
01

          
Click

          
We POST to the factory. It bakes an ee-config, runs gcloud compute instances create, waits for the VM to register.

        

        

          
02

          
Chat

          
You get redirected to chat.sm-<id>.slopandmop.com. Cookie-gated. The UI is Claude Code's TUI over ttyd; tmux keeps your session.

        

        

          
03

          
Slop

          
Say /deploy-openclaw. Claude provisions a second TDX VM, deploys ollama + openclaw, waits for post_deploy, replies with the URL.

        

        

          
04

          
Mop

          
A watchdog reaps both VMs at teardown_at. Nothing lingers. Your markdown workspace dies with the sm — that's the deal.

        

      


      

        
What's slop? An easyenclave workload JSON — a process spec plus optional
        github_release binary fetch and expose ingress. The catalog in
        slopandmop/workloads
        is the reference pile: podman-static, podman-bootstrap, nv,
        mount-models, ollama/{prod,preview}, openclaw. Your sm reads them,
        hands one to the factory, gets back a public URL. Writing your own slop is just committing another JSON.
